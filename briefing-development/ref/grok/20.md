**TLS 1.3 Key Schedule Diagram** (Simplified & Detailed)

Here is a clear overview of the **TLS 1.3 Key Derivation** process using HKDF.

### High-Level TLS 1.3 Key Schedule

```
ClientHello + ServerHello
        ↓ (ECDH or DHE)
   (EC)DHE Shared Secret
        ↓
   HKDF-Extract (with salt=0 or PSK)
        ↓
   Early Secret  ────────────────────────►  (PSK Binder Key, etc.)
        ↓
   HKDF-Extract (with (EC)DHE)
        ↓
   Handshake Secret
        ↓
   ┌──────────────────────┬──────────────────────┐
   │                      │                      │
HKDF-Expand-Label     HKDF-Expand-Label     HKDF-Expand-Label
   │                      │                      │
   ↓                      ↓                      ↓
Client HS Traffic     Server HS Traffic      Exporter Master Secret
   ↓                      ↓
Client Handshake Key   Server Handshake Key
   ↓                      ↓
   Handshake Finished messages
        ↓
   HKDF-Extract (with 0)
        ↓
   Master Secret
        ↓
   ┌──────────────────────┬──────────────────────┐
   │                      │                      │
HKDF-Expand-Label     HKDF-Expand-Label     HKDF-Expand-Label
   │                      │                      │
   ↓                      ↓                      ↓
Client App Traffic    Server App Traffic     Resumption Master Secret
   ↓                      ↓
Client Traffic Key     Server Traffic Key
```

### Detailed HKDF Steps

| Stage                    | HKDF Operation                          | Input / Context                          | Purpose |
|-------------------------|-----------------------------------------|------------------------------------------|--------|
| Early Secret            | HKDF-Extract(0, PSK or 0)              | Pre-Shared Key (if any)                 | 0-RTT support |
| Handshake Secret        | HKDF-Extract(Early Secret, (EC)DHE)    | Diffie-Hellman shared secret            | Handshake protection |
| Client/Server HS Traffic| HKDF-Expand-Label(Handshake Secret, "c hs traffic" / "s hs traffic", ...) | Transcript Hash | Handshake encryption |
| Master Secret           | HKDF-Extract(Handshake Secret, 0)      | Zero                                    | Application phase |
| Application Traffic     | HKDF-Expand-Label(Master Secret, "c ap traffic" / "s ap traffic", ...) | Transcript Hash | Application data encryption |
| Resumption Master Secret| HKDF-Expand-Label(Master Secret, "res master", ...) | - | Session resumption |

### Key Points for Your ECU Simulator

You can model your Security Access key derivation similarly:

- **Master Secret** → Base secret per ECU/OEM.
- **Seed** → Acts like the (EC)DHE shared secret (fresh challenge from ECU).
- **Label** → `kwp2000-level-${level}` or `uds-security-access`.
- **Context** → Transcript of previous messages + ECU ID.

**Simple Analogy**:
- TLS 1.3 Early Secret → Your base/master secret
- TLS 1.3 Handshake Secret → After Security Access seed exchange
- TLS 1.3 Traffic Secrets → Derived session keys for protected diagnostic commands

This design is battle-tested, provides strong security properties, and is an excellent reference for implementing robust key derivation in your KWP2000/UDS simulator.
