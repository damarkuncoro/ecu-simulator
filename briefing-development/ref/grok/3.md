**Here are detailed, byte-level frame examples** for KWP2000 (ISO 14230) on K-Line. These use common header formats for your simulator implementation.

### 1. StartCommunication (Fast Init Example)
This is often the first message after fast initialization.

**Request (Tester → ECU)**:
```
C1 33 F1 81 66
```
- `C1` — Format byte (physical addressing + length=1 data byte)
- `33` — Target address (functional often uses 0x33)
- `F1` — Source address (tester)
- `81` — Service ID (StartCommunication)
- `66` — Checksum (XOR of previous bytes: C1^33^F1^81)

**Positive Response (ECU → Tester)**:
```
83 F1 10 C1 8F D0 CS
```
- `83` — Format (length=3 data bytes)
- `F1` — Target (tester)
- `10` — Source (ECU address, e.g., Engine ECU)
- `C1` — Positive response to 0x81
- `8F D0` — Key bytes (indicate protocol capabilities, header format, timing)
- `CS` — Checksum

### 2. DiagnosticSessionControl (0x10)
Common after StartCommunication to enter Extended or Programming session.

**Request (Default → Extended Session)**:
```
C1 33 F1 10 03 CS
```
- `10` — SID DiagnosticSessionControl
- `03` — Sub-function (e.g., 0x03 = extendedDiagnosticSession)
- `CS` — Checksum

**Positive Response**:
```
83 F1 10 50 03 00 CS   (example)
```
- `50` — 0x10 + 0x40
- `03` — Echo of sub-function
- Extra bytes may include session parameters (P2/P2* timings, etc.)

**Negative Response** (example: conditions not correct):
```
83 F1 10 7F 10 22 CS
```
- `7F` — Negative response
- `10` — Original SID
- `22` — Response code (conditionsNotCorrect)

### 3. ReadDTCInformation (0x19)
Very common service.

**Request (ReportNumberOfDTCByStatusMask)**:
```
C1 33 F1 19 01 FF CS
```
- `19` — SID
- `01` — Sub-function (reportNumberOfDTCByStatusMask)
- `FF` — Status mask (all bits)

**Positive Response** (example with 2 DTCs):
```
87 F1 10 59 01 FF 02 12 34 01 56 78 03 CS
```
- `59` — Positive (0x19 + 0x40)
- `01` — Sub-function echo
- `FF` — Status mask echo
- `02` — Number of DTCs
- Then DTCs (3 bytes each: High, Low, Status)

### 4. ReadDataByIdentifier (0x22)
For reading DIDs (Data Identifiers).

**Request (e.g., read VIN or software version)**:
```
C1 33 F1 22 F1 90 CS   (example DID F190 = VIN)
```
- `22` — SID
- `F1 90` — 2-byte DID

**Positive Response**:
```
8x F1 10 62 F1 90 [Data bytes] CS
```
- `62` — 0x22 + 0x40
- Echo DID + actual data

### 5. SecurityAccess (0x27) — Seed & Key
Multi-step process.

**Step 1: Request Seed (Level 1)**:
```
C1 33 F1 27 01 CS
```
- `27 01` — SecurityAccess, requestSeed for level 1

**Positive Response (Seed)**:
```
85 F1 10 67 01 12 34 56 78 CS   (4-byte seed example)
```
- `67` — Positive (0x27 + 0x40)
- `01` — Level echo
- `12 34 56 78` — Seed value

**Step 2: Send Key**:
```
C1 33 F1 27 02 AB CD EF 01 CS   (computed key)
```
- `02` — sendKey for level 1 (usually level + 1)
- Key bytes (computed from seed using OEM algorithm)

**Positive Response (Unlocked)**:
```
83 F1 10 67 02 CS
```

**Negative** (Invalid Key): `7F 27 35` (invalidKey)

### Header Format Notes (Important for Your Parser)
- **Format Byte** bits determine:
  - Addressing type (physical/functional).
  - Whether length byte or additional length byte is present.
  - Header size (1–4 bytes common).
- Common in practice: `C1` / `81` / `83` / `84` etc.
- Checksum: Usually **XOR** of all bytes before it (confirm with key bytes from init).

### Tips for Your ECU Simulator
- Support configurable **header modes** and addresses.
- Implement **response pending (0x78)** for long operations.
- Add **P2/P3 timer** delays with jitter in `timing-engine`.
- Make checksum calculation/validation pluggable.
- Log raw bytes + decoded interpretation in your CLI/logger.
