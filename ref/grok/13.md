**Full Game-Based Security Proof for HKDF** (Standard Cryptographic Style)

Here is a **game-based proof** outline for HKDF’s security, following the standard approach used in Krawczyk’s analysis and modern cryptographic papers.

### Security Definition (KDF Game)

We define a game between a **Challenger** and an **Adversary** $\mathcal{A}$:

- **Setup**: Challenger samples a random master secret `IKM` (or uses adversary-chosen IKM in some variants).
- **Oracle Access**: $\mathcal{A}$ can query `HKDF(salt, info, length)` any number of times.
- **Challenge**: Challenger flips a bit $b \in \{0,1\}$:
  - If $b=0$: Return real HKDF-derived key.
  - If $b=1$: Return a uniformly random string of the same length.
- **Guess**: $\mathcal{A}$ outputs $b'$. It wins if $b' = b$.

**Advantage**:
$$
\text{Adv}^{\text{KDF}}(\mathcal{A}) = |Pr[b' = b] - 1/2|
$$
HKDF is secure if this advantage is **negligible** for any efficient $\mathcal{A}$.

---

### Game-Hopping Proof

**Game 0** (Real Game):  
The real HKDF game as defined above.  
$\text{Adv}_0 = \text{Adv}^{\text{KDF}}(\mathcal{A})$

**Game 1**: Replace the **Extract** phase with a truly random PRK.  
Instead of `PRK = HMAC(salt, IKM)`, the challenger samples `PRK` uniformly at random.

**Difference**:  
$|Pr[\text{Game 1}] - Pr[\text{Game 0}]| \leq \text{Adv}^{\text{PRF-HMAC}}(\mathcal{B})$

**Reasoning (Reduction)**:  
If $\mathcal{A}$ can distinguish Game 0 from Game 1, we can build a PRF distinguisher $\mathcal{B}$ against HMAC:
- $\mathcal{B}$ uses its PRF oracle as the HMAC in Extract.
- When $\mathcal{A}$ queries HKDF, $\mathcal{B}$ forwards the salt/IKM to its oracle to get PRK, then proceeds with Expand.
- If the oracle is real HMAC → Game 0 behavior.
- If the oracle returns random → Game 1 behavior.

Thus, a non-negligible difference implies breaking HMAC-PRF security.

**Game 2**: In the **Expand** phase, replace each HMAC call with a random function (or random output).

**Difference**:  
$|Pr[\text{Game 2}] - Pr[\text{Game 1}]| \leq q \cdot \text{Adv}^{\text{PRF-HMAC}}(\mathcal{C})$  
(where $q$ is the number of Expand queries)

**Reasoning**:  
Each $T_i = \text{HMAC}(PRK, \text{prefix} || \text{info} || \text{counter})$ is replaced by random.  
Since PRK is random (from Game 1) and HMAC is a PRF, each output is indistinguishable from random. The chaining and counter provide domain separation.

**Game 3** (Ideal Game):  
All outputs are purely random strings (independent of IKM).

In this game, the adversary’s advantage is exactly **0** (pure random vs random is indistinguishable).

---

### Overall Security Bound

$$
\text{Adv}^{\text{HKDF}}(\mathcal{A}) \leq \text{Adv}^{\text{HMAC-PRF}}(\mathcal{B}) + q \cdot \text{Adv}^{\text{HMAC-PRF}}(\mathcal{C})
$$

Where:
- The first term comes from the Extract reduction.
- The second term bounds the Expand phase (multiplied by number of queries).

This bound is **very strong** when using HMAC-SHA256.

### Key Proof Techniques Used

- **Game Hopping** — Incremental changes, each bounded by a reduction to a primitive (HMAC-PRF).
- **PRF Security** — Core assumption.
- **Domain Separation** — The `info` string + counter prevents collisions between different derivations.
- **Random Oracle-like Behavior** — In some analyses, the hash is modeled as a random oracle for simplicity.

### Practical Takeaways for Your Simulator

- When you implement full HKDF (Extract + Expand with proper `info` context including level + seed), you inherit this strong security reduction.
- The proof justifies using HKDF as your **strong/default** algorithm in the `security-engine`.
- Simpler algorithms (XOR, linear) have **no such formal proof** — they rely on obscurity.
