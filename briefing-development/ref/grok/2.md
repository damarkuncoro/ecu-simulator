**KWP2000 (Keyword Protocol 2000)** is the diagnostic protocol standardized in **ISO 14230** (parts 1–4). It is widely used on **K-Line** (ISO 9141-2 compatible) for ECU communication in many vehicles (especially European and Asian models from the late 90s to mid-2010s).

It bridges older ISO 9141 and modern **UDS (ISO 14229)** protocols.

### 1. Protocol Layers (ISO 14230)
- **Part 1**: Physical Layer (K-Line, electrical specs, baud rates).
- **Part 2**: Data Link Layer (framing, timing, error handling).
- **Part 3**: Implementation of Diagnostic Services (based on ISO 14229).
- **Part 4**: Requirements for emission-related systems (OBD compliance).

### 2. Physical Layer & Initialization
K-Line is a single-wire bidirectional line (usually pin 7 in OBD-II connector). Optional L-Line (pin 15) for some older 5-baud inits.

**Two main initialization methods**:

**A. Slow Initialization (5-baud init)** — Classic method:
1. Tester sends target address (e.g., `0x33` for functional or specific ECU address) at **5 baud** (very slow — takes ~2 seconds per byte).
2. ECU responds with synchronization pattern `0x55`.
3. ECU sends two **Key Bytes** (KB1, KB2) indicating supported features and baud rate (commonly 10400 baud).
4. Tester echoes back KB2 inverted.
5. ECU sends address byte inverted.
6. Communication continues at normal baud rate (usually 10400 bps).

**B. Fast Initialization** (preferred in KWP2000):
- Tester pulls K-Line low for 25 ms, then high for 25 ms.
- Immediately starts sending at 10400 baud.
- Common first frame: `StartCommunication` request (e.g., `C1 33 F1 81 66`).

Your simulator should support **both**, with configurable timing/jitter.

### 3. Frame Format (Data Link Layer)
KWP2000 uses variable-length frames with these elements (order depends on header mode):

- **Format Byte** (always present) — Defines addressing, length type.
- **Target Address** (optional).
- **Source Address** (optional).
- **Length** (1 or 2 bytes).
- **Service ID (SID)** + parameters.
- **Checksum** (XOR or sometimes other) — last byte.

**Common Header Modes**:
- Physical addressing (targeted ECU).
- Functional addressing (broadcast to group).

**Example frames** (simplified):
- Request: `[Header] [SID] [Params] [Checksum]`
- Positive response: SID + 0x40 (e.g., 0x10 → 0x50).
- Negative response: 0x7F + original SID + Response Code.

| SID (Hex) | Service Name                    | Positive Resp | Common Use                              | Notes |

| 0x10     | DiagnosticSessionControl       | 0x50         | Switch sessions (default, extended, programming) | Critical for unlocking features |

Negative responses use **0x7F** + error codes (e.g., 0x11 = Service Not Supported, 0x12 = Subfunction Not Supported, 0x78 = Request Correctly Received - Response Pending).

### 5. Timing Parameters (Critical for Compliance)
- **P1**: Inter-byte time in a message.
- **P2**: Time between request and response.
- **P2***: Extended for long operations.
- **P3**: Time between responses or tester requests.
- **P4**: Inter-byte in ECU response.

Your `timing-engine` package should simulate realistic jitter and respect these for HIL testing.

### 6. Implementation Tips for Your Simulator
- **State Machine**: Implement session management (default vs extended), security levels, and Px timers.
- **Fault Injection**: Timeout simulation, checksum errors, delayed responses, invalid SIDs.
- **DID Registry**: Strongly typed Data Identifiers with validation.
- **OEM Variants**: Many manufacturers deviate slightly (baud rate, init sequence, custom SIDs) — your OEM profiles folder is a great idea.
- **Checksum**: Usually simple XOR of all bytes.
- **Multi-ECU**: Support functional addressing and multiple virtual ECUs.

**KWP2000 vs UDS**:
- KWP2000 is the predecessor.
- UDS (ISO 14229) reuses many services but is cleaner and primarily on CAN (ISO 15765).
- Many ECUs transitioned from KWP on K-Line to UDS on CAN.
