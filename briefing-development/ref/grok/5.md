**Security Access (0x27) Algorithms in KWP2000**

The **Security Access** service is one of the most important (and manufacturer-specific) parts of KWP2000. It protects sensitive operations such as flashing, coding, clearing DTCs, or writing parameters. The standard defines the message flow, but **the actual seed-to-key transformation is proprietary** to the OEM or ECU supplier (Bosch, Siemens, Delphi, etc.).

### Standard Message Flow (0x27)
1. **Request Seed** (sub-function = odd number, usually `01`, `03`, `05`...):
   ```
   [Header] 27 01 [CS]
   ```

2. **ECU Response** (positive):
   ```
   [Header] 67 01 [Seed bytes (1-16 bytes typical)] [CS]
   ```
   Example: `67 01 12 34 56 78` (4-byte seed)

3. **Send Key** (sub-function = request + 1, usually `02`, `04`...):
   ```
   [Header] 27 02 [Key bytes] [CS]
   ```

4. **ECU Response**:
   - Success: `67 02 [CS]`
   - Failure: `7F 27 35` (invalidKey) or `36` (exceededNumberOfAttempts)

After success, the ECU unlocks the requested security level for the current session.

### Common Seed-Key Algorithms

These are **not** part of the ISO standard — they are implemented in the ECU firmware.

**1. Simple XOR (Very Common)**
- Key = Seed XOR fixed_value (often per level)
- Example: `key = seed ^ 0x12345678` (for 4-byte values)
- Or byte-wise XOR with a secret byte array.

**2. Bit Manipulation / Rotation + XOR**
- Rotate left/right, invert bits, then XOR.
- Example (2-byte from SIMK43/41):
  ```python
  def calculate_key(seed):  # seed as int
      key = 0x9360
      for _ in range(0x24):   # 36 iterations
          key = (key * 2) ^ seed
      return key & 0xFFFF
  ```

**3. Linear Congruential / Multiplicative**
- `key = (seed * multiplier + adder) % 2^16` (or 2^32)
- Common in many Bosch ECUs.

**4. 2's Complement / Negation**
- Key = ~seed + 1 (or simple inversion).

**5. More Complex (Modern ECUs)**
- AES-128/256 encryption of the seed (with secret key).
- Hash-based (e.g., simple CRC or custom).
- Time-dependent or counter-based (seed includes a challenge that changes).

**6. OEM-Specific Examples**
- **VAG (Volkswagen Group)**: Uses multiple levels; algorithms often in dealer tools (ODIS, VCDS). Some are simple XOR or table-based.
- **Bosch ME7 / MED17**: Varies by software version; many use XOR or polynomial.
- **Delphi / Siemens**: Often rotation + XOR.

### Implementation in Your Simulator (`security-engine`)

**Recommended Architecture**:
- **Plugin System** — Allow loading OEM-specific algorithms (JS/TS functions or classes).
- **Registry** — Map ECU type / supplier / level → algorithm.
- **Default Fallbacks** — Include common ones (XOR with configurable mask, linear, etc.).
- **Seed Generation** — ECU should generate fresh random or pseudo-random seeds on each request (to prevent replay attacks).

**Example TypeScript Structure**:
```ts
interface SecurityAlgorithm {
  getKey(seed: Buffer, level: number, context?: any): Buffer;
}

class XorAlgorithm implements SecurityAlgorithm {
  constructor(private mask: Buffer) {}
  getKey(seed: Buffer): Buffer {
    return Buffer.from(seed.map((b, i) => b ^ this.mask[i % this.mask.length]));
  }
}

// Plugin registration
securityEngine.register('bosch-me7', 1, new XorAlgorithm(Buffer.from([0x55, 0xAA,...])));
```

### Best Practices for Your ECU Simulator
- Make seed length and algorithm **configurable per virtual ECU**.
- Support **delay / attempt counter** (lockout after N wrong keys).
- Allow **bypass mode** for testing (always accept key or fixed seed/key pair).
- Log seed → expected key for debugging.
- Support multiple simultaneous security levels.
- Add fault injection: wrong key response, delayed response, seed replay protection.

**Testing Tips**:
- Test with known good seed/key pairs from real logs.
- Verify negative responses (`35`, `36`, `37` — requiredTimeDelayNotExpired, etc.).
- Ensure session timeout resets security access.
