**HKDF Security Proofs & Properties**

**HKDF** (HMAC-based Extract-and-Expand Key Derivation Function), defined in [RFC 5869](https://datatracker.ietf.org/doc/html/rfc5869), is one of the most rigorously analyzed and trusted key derivation functions in modern cryptography.

### Core Security Guarantees

1. **Pseudorandomness**
   - If the input keying material (IKM) has sufficient min-entropy, the output is **indistinguishable from random**.
   - Even if IKM has low entropy (e.g., a password or fixed master secret), HKDF still produces strong outputs when used correctly.

2. **Key Extraction Security**
   - The **Extract** phase turns arbitrary-length, possibly weak IKM into a fixed-length, high-entropy **PRK** (Pseudorandom Key).
   - Proof basis: HMAC is a secure pseudorandom function (PRF) when the key is fixed and the input varies.

3. **Key Expansion Security**
   - The **Expand** phase produces multiple output keys that are computationally independent.
   - Knowing one derived key does **not** help recover the PRK or other derived keys (domain separation via the `info` string and counter).

### Formal Security Model

HKDF is proven secure in the **Random Oracle Model** (ROM) and under standard assumptions about HMAC:

- **PRF Security** — HMAC-SHA256 is a secure PRF (proven under the assumption that SHA-256 compression is a PRF).
- **Indistinguishability** — The derived keys are indistinguishable from uniform random strings to an adversary who does not know the master secret.
- **Multi-Stage Security** — HKDF supports "keying material" that comes from previous stages (e.g., Diffie-Hellman shared secret) securely.

**Key Papers & Proofs**:
- **Krawczyk (2006/2010)** — Original HKDF design and analysis by Hugo Krawczyk (creator of HMAC).
- **Bellare et al.** — Analysis of HMAC as a PRF.
- HKDF is used in TLS 1.3, Noise Protocol, Signal, etc., and has withstood extensive cryptanalysis.

### Practical Security Properties Relevant to ECU Diagnostics

| Property                        | Benefit for Security Access (0x27)                          | Strength |
|--------------------------------|-------------------------------------------------------------|----------|
| Domain Separation              | Different `info` + level → different keys                  | Excellent |
| Resistance to Length Extension | HMAC prevents certain hash-based attacks                   | Excellent |
| Salt Usage                     | Protects against rainbow tables / precomputation           | Good |
| Forward Secrecy (with fresh seed) | New seed each time → new derived key                     | Strong |
| Resistance to Key Reuse        | Different levels produce isolated keys                     | Strong |

### Limitations & Attack Considerations

- **If Master Secret is Compromised** → All derived keys are compromised (standard for any KDF).
- **Weak Randomness in Seed** → HKDF mitigates this via Extract phase, but strong randomness is still recommended.
- **No Built-in Authentication** → HKDF is a KDF, not a MAC. Use CMAC or HMAC on the final key if needed.
- **Side-Channel Attacks** → Timing/power analysis on HMAC implementation (relevant for real ECUs).
