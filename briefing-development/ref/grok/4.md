**Full Typical KWP2000 Diagnostic Session Sequence**

Here is a complete, realistic end-to-end sequence for a KWP2000 session on K-Line (commonly used for engine, airbag, ABS, etc. ECUs). This is what your simulator should support and test.

### Phase 0: Physical Connection & Wake-up
- Connect to K-Line (and optionally L-Line).
- Optional: Wake-up pattern (some ECUs need K-Line toggled).

### Phase 1: Initialization (Mandatory)
**Option A: Fast Initialization (most common in KWP2000)**
1. Tester pulls K-Line **low for 25 ms**, then **high for 25 ms**.
2. Immediately send **StartCommunication** request at 10400 baud.

**StartCommunication Request**:
```
C1 33 F1 81 66
```

**ECU Positive Response** (example):
```
83 F1 10 C1 8F D0 XX
```
(Key bytes `8F D0` indicate header format, timing params, etc.)

**Option B: Slow (5-baud) Initialization**
- Send address byte (e.g. `0x33`) at **5 baud**.
- ECU replies with `0x55` (sync), key bytes, etc.

After successful init → communication is established at 10400 baud (or as negotiated).

### Phase 2: Start Diagnostic Session
Send **DiagnosticSessionControl** to enter a non-default session.

**Request (Enter Extended Diagnostic Session)**:
```
C1 33 F1 10 03 CS
```
- `10` = SID
- `03` = extendedDiagnosticSession

**Positive Response**:
```
83 F1 10 50 03 PP PP CS
```
(`50` = positive, parameters may include timing values)

**Common sessions**:
- `01` — defaultSession
- `02` — programmingSession
- `03` — extendedDiagnosticSession

### Phase 3: Tester Present (Keep Session Alive)
Periodically send (every ~3–5 seconds):
```
C1 33 F1 3E 00 CS   (or without sub-function)
```
ECU usually replies with positive `7E`.

### Phase 4: Security Access (If Required)
Many operations (clear DTCs, write data, flashing) need this.

**1. Request Seed (Level 1)**:
```
C1 33 F1 27 01 CS
```

**ECU Response (Seed)**:
```
85 F1 10 67 01 AA BB CC DD CS   (4-byte seed example)
```

**2. Compute Key** (OEM-specific algorithm, e.g., XOR with fixed value, bit manipulation, or seed-key table).

**Send Key**:
```
C1 33 F1 27 02 [Key bytes] CS
```

**ECU Success**:
```
83 F1 10 67 02 CS
```
Security level 1 is now unlocked.

### Phase 5: Perform Diagnostics (Common Operations)

**A. Read DTCs**
```
C1 33 F1 19 02 FF CS   (reportDTCByStatusMask)
```
ECU returns number of DTCs + list (DTC + status).

**B. Clear DTCs**
```
C1 33 F1 14 FF FF FF CS   (ClearDiagnosticInformation)
```

**C. Read Data (Sensors, Identifiers)**
```
C1 33 F1 22 F1 90 CS   (ReadDataByIdentifier — e.g. VIN)
```

**D. ECU Reset**
```
C1 33 F1 11 01 CS   (hardReset)
```

**E. Other Common Services**
- `2E` — WriteDataByIdentifier
- `31` — RoutineControl (e.g. self-test)
- `22` / `23` — Read memory / data

### Phase 6: End Session / Return to Default
- Send `10 01` (defaultSession).
- Or just stop communication (`82 33 F1 82 CS` for StopCommunication).

### Full Example Sequence (Byte Level Summary)

1. Fast Init + `C1 33 F1 81 66` → Response
2. `C1 33 F1 10 03 CS` → Session extended
3. `C1 33 F1 27 01 CS` → Seed
4. `C1 33 F1 27 02 [Key] CS` → Unlocked
5. `C1 33 F1 19 02 FF CS` → Read DTCs
6. `C1 33 F1 14 FF FF FF CS` → Clear DTCs (if security allows)
7. `C1 33 F1 3E CS` → TesterPresent (repeated)
8. `C1 33 F1 10 01 CS` → Back to default

### Implementation Notes for Your Simulator
- **State Machine** (in `session-fsm`): Track current session, security level, communication state.
- **Timeouts**: Enforce P2, P3, S3 (session timeout) — very important.
- **Addressing**: Support both physical (specific ECU) and functional (0x33).
- **Negative Responses**: Implement common codes (`11`, `12`, `22`, `31`, `35`, `78`, etc.).
- **Fault Injection**: Drop frames, delay responses, send wrong checksum, etc.
- **OEM Variations**: Some ECUs skip security for basic read, use different key algorithms, or custom DIDs.

This sequence is what real diagnostic tools (like OBD scanners, dealer tools) follow.
