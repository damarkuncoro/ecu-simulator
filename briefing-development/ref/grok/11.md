**HKDF Formal Proof Details** (Accessible yet Rigorous Overview)

**HKDF** was introduced by **Hugo Krawczyk** in 2010. Its security is formally analyzed in the cryptographic literature. Here are the key formal results:

### 1. Main Security Theorem (from Krawczyk's Paper)

**Theorem (Simplified)**:  
If HMAC is a secure **pseudorandom function (PRF)**, then HKDF is a secure **variable-length output key derivation function** in the **Random Oracle Model** (and under standard assumptions even in the standard model for HMAC).

**Two main properties proven**:

- **Indistinguishability** — The output of HKDF is computationally indistinguishable from a truly random string of the same length to an adversary who does **not** know the input keying material (IKM).
- **Key Independence** — Different calls with different `info` strings (or counters) produce outputs that are independent, even if the underlying PRK is the same.

### 2. Security Model

HKDF is proven secure in the following model:

- **Adversary** can query the derivation oracle many times with chosen salts, info strings, and IKM.
- **Goal** — Distinguish the derived key from random.
- **Assumption** — The underlying hash function (used in HMAC) behaves like a random oracle or satisfies PRF properties.

**Core Reduction**:
- Security of HKDF reduces to the security of **HMAC** as a PRF.
- HMAC-SHA256 is widely believed to be a secure PRF (no practical breaks known as of 2026).

### 3. Extract Phase Proof Sketch

**Extract** turns possibly weak IKM into a strong PRK.

**Security Argument**:
- HMAC(salt, IKM) is a secure PRF when salt is fixed (or random).
- Even if IKM has low entropy, as long as there is some unpredictability, the output PRK is close to uniform.

This is why HKDF can safely derive keys from Diffie-Hellman shared secrets, passwords (with salt), or fixed master secrets.

### 4. Expand Phase Proof Sketch

**Expand** uses the PRK as a key to HMAC in a chain:

```
T(1) = HMAC(PRK, info || 0x01)
T(2) = HMAC(PRK, T(1) || info || 0x02)
...
```

**Security**:
- Each T(i) is pseudorandom because HMAC is a PRF.
- The chaining ensures that knowing one T(i) does not help predict T(j) for j ≠ i (due to the counter and prefix).
- This gives **domain separation** and **key independence**.

### 5. Important Security Bounds

- **Advantage** of adversary is roughly bounded by:
  - Number of queries × advantage against the underlying HMAC.
- Very strong concrete security when using SHA-256 or SHA-512.

### 6. Known Limitations (Honest Discussion)

- **Random Oracle Assumption** — The proofs assume the hash function is ideal. Real hashes (SHA-256) are not perfect random oracles, but no practical attacks exploit this for HKDF.
- **Master Secret Compromise** — If the top-level IKM/master secret leaks, all derived keys are compromised (standard limitation of any KDF).
- **Side Channels** — Proofs are for black-box crypto; timing, cache, or power attacks on the implementation are out of scope.
- **No Forward Secrecy by Default** — You must use fresh seeds or ephemeral keys to get PFS.
